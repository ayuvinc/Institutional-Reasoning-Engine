Build Your Own Institutional Reasoning Engine

A complete guide for non-technical founders to understand, plan, and build an IRE — the AI system that makes investigation outputs defensible, auditable, and traceable to evidence. Every tool, every subscription, every workflow explained in plain English.

What Is an Institutional Reasoning Engine?

An IRE is not a chatbot. It is not an automation tool. It is an AI system that runs a structured investigation workflow — forming a hypothesis, iterating through evidence, and requiring a human to approve every finding before it enters the record.

The One-Sentence Version

An IRE is a professional investigation workflow encoded in software — AI accelerates the analysis, humans remain accountable for every conclusion.

The Three Things That Make It Different

🔒 Methodology as Engineering

Investigation rules are baked into the system — not left to human discretion. The AI cannot skip a step any more than a calculator can divide by zero.

👤 Human Gate at Every Finding

No AI output enters the official record without a named investigator approving it. The approval IS the record — not a step before the record.

🔗 Tamper-Evident Audit Chain

Every action is cryptographically linked. If anyone alters a log entry — even one character — every subsequent entry breaks. Mathematically detectable.

The 7 Modular Clusters

Cluster A

Documentary Evidence

Turns any document pile into a searchable evidence base. PDFs, spreadsheets, emails — all queryable by meaning.

Cluster B

Behavioural & Interview

Detects contradictions between interview statements and documentary evidence. Cites both sides automatically.

Cluster C

Entity & Network Intelligence

Maps who is connected to whom. Surfaces circular flows and fraud rings invisible from individual documents.

Cluster D

Advanced Reasoning

The recursive reasoning engine. Forms hypotheses, tests them, revises them, repeats until confident or flagged.

Cluster E

Privacy Gateway

Replaces real identifiers with tokens before anything reaches external AI. De-tokenises only on approval.

Cluster F

Institutional Memory

Fine-tunes the AI on your own completed cases. The system learns your fraud typologies over time.

Cluster G

Integrity Elevation

Daily external anchoring of the audit chain. Elevates from institutional-grade to court-grade evidence integrity.

Prerequisites

Before you write a single line of code or sign up for any tool, three things must be true. These are not optional — the system encodes your methodology, and if no methodology exists, there is nothing to encode.

Do This First

Most teams skip these prerequisites and then wonder why the AI produces irrelevant or inconsistent outputs. The IRE encodes your investigation methodology — if that methodology is not written down, the build will fail at the hardest moment.

1

Document Your Investigation Methodology

Write down exactly how your investigations work today: how a case is opened, what evidence types you collect, what constitutes a finding, what your output documents look like (STR, HLIR, case closure memo, etc.). This becomes the specification the system is built against. Use Claude or ChatGPT to help draft this if you don't have it written yet — describe your process in conversation and ask it to structure it as a methodology document.

2

Complete a Data Inventory

List every type of evidence you routinely handle: bank statement CSVs, KYC PDFs, email exports, interview recordings, transaction logs, corporate registry documents. For each type, note: what format it comes in, where it is stored today, what personal data it contains. This determines which clusters you need and shapes your Cluster E (privacy) design.

3

Get Legal & Compliance Sign-Off

Before any AI touches investigation data, your legal and compliance team must sign off on three things: (a) human review gate requirements in your jurisdiction, (b) whether pseudonymisation is sufficient for your data protection obligations, (c) whether AI-generated outputs are admissible in your regulatory proceedings. This is not a day-one problem — but it must be resolved before Phase 6 (report generation) goes live.

Cost Optimisation Tips

Use spot/preemptible instances for GPU compute — 60–70% cheaper. Only run the 70B model when needed — use 8B for orchestration tasks. Batch SOTA API calls — one call per section, not one per claim. Cache embeddings — documents don't need re-embedding unless they change.

Phase 1 — Mandatory Base (Weeks 1–6)

Deliverable

A working case management system with tamper-evident audit logging and human approval gates. Defensible output, even with no AI reasoning yet.

What to Build

  1. Case creation and management (PostgreSQL + simple API)
  2. Case-scoped namespaces in Qdrant (B1)
  3. Audit event logger writing to PostgreSQL (B5)
  4. Hash-chain implementation — SHA-256 chaining on every log entry (B6)
  5. Human review gate — approval UI that writes to audit chain (B4)
  6. Model version pinning — store model_id + model_version with every AI call (B7)
  7. Bias monitoring dashboard — basic demographic flag rate tracker (B8)

Exit Criteria

  • 10 synthetic cases run end-to-end with complete audit chains
  • Hash chain verification script confirms no tampering detectable
  • Human review gate blocks record creation without approval event
  • Zero cross-case data leakage confirmed by namespace isolation test

AI Tools to Use for This Phase

  • Claude Code / Cursor: Write the PostgreSQL schema, hash-chain implementation, and basic API. Prompt: "Build a PostgreSQL-backed audit logger with SHA-256 hash chaining where each entry links to the previous one. Each entry has: event_id, timestamp, event_type, payload, payload_hash, chain_hash."
  • Claude (this interface): Review architecture decisions, write test cases, design the UI wireframes for the human review gate.

Phase 2 — Cluster A: Document Ingestion (Weeks 7–14)

Deliverable

Investigators can upload any document type and query it by meaning. Evidence retrieval in under 1 second. Every claim cites its source document and page.

What to Build

  1. Document parser — PyMuPDF for PDFs, openpyxl for Excel, python-docx for Word
  2. Document-type-aware chunker — different strategies per file type
  3. Local embedding model — BGE-M3 via Ollama
  4. Qdrant indexer — writes to case-scoped namespace
  5. Retrieval API — semantic search with namespace filter
  6. Retrieval Verifier — checks claims against namespace (B2)
  7. Evidence Grounder — attaches chunk_id + source + page (B3)

Exit Criteria

  • 50 real documents ingested across 5 different file types
  • Retrieval latency under 1 second post-index
  • Retrieval Verifier correctly strips 100% of injected false claims in test
  • Evidence Grounder attaches correct citations to all verified claims

Phase 3 — Cluster B: Behavioural & Parallel Run (Months 4–9)

Deliverable

Interview transcripts ingested and searchable. Contradiction detection live. Parallel run: new system runs alongside old process — outputs reviewed but not primary record.

What to Build

  1. Whisper STT integration — local transcription with speaker diarisation
  2. Transcript chunker — speaker-turn level, with timestamp + speaker metadata
  3. Extended PII recognisers for Presidio — spoken-form names, informal references
  4. Contradiction Detector — cross-references transcript claims vs document corpus and graph
  5. Transcript Approval Gate — separate HITL approval flag for statement-derived findings

Phase 4 — Clusters C + D: Graph + Reasoning (Month 4–6)

Deliverable

Entity network mapping live. Recursive reasoning loop operational. Evidence scoring running. Profile 4 capability achieved.

What to Build

  1. Entity extractor — NER pipeline feeding into resolution queue
  2. Three-tier entity resolution — Tier 1 auto, Tier 2/3 blocking HITL screens
  3. Neo4j graph builder — entities as nodes, transactions as weighted edges
  4. Graph query layer — circular flow, community detection, path analysis
  5. LangGraph reasoning loop — Planner → Analyzer → Verifier → Grounder → Critic → Score
  6. Hypothesis Tracker — maintains working theory across iterations
  7. Calibrated evidence scorer — 4-component deterministic score

Phase 5 — Clusters E + G: Privacy + Integrity (Month 6–10)

Legal Sign-Off Required Before This Phase

Cluster E activates external AI APIs. Get legal review of your pseudonymisation approach and data processing agreements before enabling.

What to Build

  1. Presidio pseudonymisation pipeline with custom recognisers
  2. Token map store — in-memory, encrypted at rest, session-scoped
  3. De-tokenisation gate — restores identifiers only post-approval, on-premise
  4. External SOTA API integration (Claude or GPT-4) — receives pseudonymised payload only
  5. G1 — AWS QLDB / Azure Confidential Ledger integration for daily audit chain anchoring
  6. G2 — Recording file hash → transcript linkage in audit chain

Phase 6 — Report Generation + Primary System (Month 10+)

Deliverable

Full draft report generation in 60–90 seconds. Every claim cited. Human review protocol live. New system becomes primary investigation record.

What to Build

  1. Query template library — one RAG + graph query pair per report section
  2. Parallel section generator — all 8 sections generated simultaneously
  3. Section review UI — finding + citation map side by side, edit + approve interface
  4. Citation Map appendix — auto-generated index of all findings to sources
  5. Final report lock — approved sections assembled, report hash stored in audit chain

Critical: Do Not Skip Entity Resolution

A 10% entity duplication rate in a 500-node graph produces 50 phantom nodes — enough to break circular flow detection entirely. This is the gap that most investigation AI systems fail to address honestly. Build Tier 2/3 resolution before building the graph layer.

Known Limitation — Inference Re-Identification

Pseudonymisation controls what enters the AI — it does not control what the AI infers. A model given pseudonymised data may produce output that, combined with other information, re-identifies a data subject. In Plain English

Prompting Principles for IRE Build

1

Specify Inputs and Outputs Exactly

Don't say "build a chunker." Say "Build a chunker that takes a string of PDF text and returns a list of dicts with keys: chunk_id (UUID), text (string), page_ref (int), source_doc (string), char_count (int). Use 500-char chunks with 100-char overlap."

2

Include the Constraint

Always add the non-negotiable architectural constraint: "The namespace_id must match the case_id — never query across namespaces" or "Temperature must be 0 — no randomness in extraction tasks."

3

Ask for Tests Alongside Code

Add "also write 3 pytest test cases" to every code request. Test cases clarify what the code should do and catch errors before they reach production.

4

Review Before Running

Read the generated code before running it in production. You don't need to understand every line — but you should understand what it does. Ask Claude to explain any section you don't follow.

5

Keep a Decision Log

Every architectural decision you make — "we chose Qdrant over Pinecone because of data residency" — write it down. This becomes your institutional knowledge and makes onboarding new developers faster.

Frequently Asked Questions

Do I need to know how to code to build this?▶

You need at least one person who can write Python and work with APIs. With Claude Code or Cursor, a single developer can build Profile 3 in 8–12 weeks. You as the founder provide the domain knowledge — what the system needs to do, what constitutes a finding, what the output should look like. AI tools handle most of the actual code writing.

Can I use cloud services instead of self-hosting everything?▶

Yes, but with a critical caveat: investigation data is sensitive. The vector database (Qdrant), graph database (Neo4j), audit log (PostgreSQL), and PII token map must stay on-premise or in a jurisdiction-compliant private cloud. Only pseudonymised data may reach external AI APIs (Cluster E). For Profiles 1–4, everything can run on a private cloud VM without any external AI.

How long before the system delivers value?▶

Phase 1 (Mandatory Base) delivers defensible audit-logged case management in 4–6 weeks — before any AI reasoning is active. Phase 2 (document retrieval) delivers evidence search in under 1 second by week 14. Phase 4 (recursive reasoning) delivers draft findings in 60–90 seconds. You don't need the full platform to start getting value.

What is the minimum hardware I need?▶

For Profiles 1–3 (no 70B model): a cloud VM with 8GB RAM and 4 vCPUs is sufficient (~$30–50/month on AWS/GCP). For Profile 4+ (70B model): you need a GPU instance with at least 40GB GPU RAM. AWS g5.12xlarge or GCP a2-highgpu-1g. Budget $150–400/month for compute. Alternatively, use Ollama on a powerful workstation (RTX 4090 = 24GB VRAM — run 2 instances for 70B).

Is the system admissible in court/regulatory proceedings?▶

The architecture is designed for maximum defensibility — tamper-evident audit chain, human approval at every finding, evidence citations on every claim. With Cluster G active (external notarisation), integrity is independently verifiable. However, admissibility is a legal question that depends on jurisdiction, proceeding type, and regulatory requirements. Engage qualified legal counsel in your jurisdiction before relying on IRE outputs in any formal proceeding.

How do I handle Indian regulatory requirements (RBI, PMLA, SEBI)?▶

The architecture is jurisdiction-agnostic — map your specific regulatory framework into the investigation methodology document (Prerequisite 1). For RBI/PMLA: your STR templates, threshold amounts, and filing windows become the query template library (A5) and framework coverage checklist (D4). For data residency: all data stores must be on Indian infrastructure — use AWS Mumbai, Azure India, or domestic providers (Jio Cloud, Tata TCS). Engage a FEMA/PMLA-specialist counsel to sign off on the AI-in-investigation workflow before production.

When should I NOT build this?▶

Don't build this if: (1) you have fewer than 5 investigations per month — the overhead exceeds the value at that scale; (2) you don't have documented investigation methodology — the system has nothing to encode; (3) your investigations are entirely unstructured conversations with no defined output format; (4) you cannot allocate one developer for 3+ months. In these cases, start with a simpler case management system and return to IRE when volume and complexity justify it.

What is the difference between institutional-grade and court-grade integrity?▶

Institutional-grade (Base B5+B6): the audit chain is tamper-evident — any modification breaks the cryptographic links, detectable by anyone with access to your systems. Court-grade (+ Cluster G): the daily chain hash is anchored to an independent external ledger. A third party can verify integrity without accessing your systems at all. "We say it wasn't altered" vs "a third party can prove it wasn't altered." Cluster G is only needed when outputs will face adversarial legal challenge.
