What Is an Institutional Reasoning Engine?

An IRE is not a chatbot. Not automation. It is an AI system that runs a structured investigation workflow — forming a hypothesis, iterating through evidence, and requiring human approval at every finding before it enters the record.

The One-Sentence Version

A professional investigation workflow encoded in software — AI accelerates the analysis, humans remain accountable for every conclusion.

The Three Things That Make It Different

🔒 Methodology as Engineering

Investigation rules are baked into the system — not left to human discretion. The AI cannot skip a step any more than a calculator can divide by zero.

👤 Human Gate at Every Finding

No AI output enters the official record without a named investigator approving it. The approval IS the record — not a step before the record.

🔗 Tamper-Evident Audit Chain

Every action is cryptographically linked. Alter one log entry and every subsequent entry breaks. Mathematically detectable.

Prerequisites

Three things must be true before you write a single line of code. The system encodes your methodology — if it is not documented, the build will fail at the hardest moment.

Do This First

Most teams skip prerequisites and then wonder why the AI produces inconsistent outputs. The IRE encodes your investigation methodology — if that methodology is not written down, there is nothing to encode.

1

Document Your Investigation Methodology

Write down exactly how investigations work today: how a case is opened, what evidence types you collect, what constitutes a finding, what your output documents look like (STR, HLIR, case closure memo). This becomes the specification the system is built against. Use Claude to help draft this — describe your process and ask it to structure it as a methodology document.

2

Complete a Data Inventory

List every type of evidence you routinely handle: bank statement CSVs, KYC PDFs, email exports, interview recordings, transaction logs. For each type note: format, where stored, what personal data it contains. This determines which clusters you need and shapes your Cluster E design.

3

Get Legal & Compliance Sign-Off

Before AI touches investigation data, your legal team must sign off on: (a) human review gate requirements in your jurisdiction, (b) whether pseudonymisation is sufficient for your data protection obligations, (c) whether AI-generated outputs are admissible in your regulatory proceedings.

The Real Implementation Challenge: Data Messiness

What Actually Dominates Months 1–3

AI reasoning failures are not your biggest early problem. Data quality is. Plan for this from day one — not as an afterthought when investigators complain that documents are "not showing up."

The Most Common Production Blockers

ProblemWhat HappensFix
Scanned PDFs without OCR layerPyMuPDF returns empty string. Ingestion pipeline produces empty chunks. Evidence silently missing from case.Detect text layer presence before parsing. If empty → route to Tesseract automatically. Never fail silently.
Password-protected filesParser throws cryptic error. File never ingests. Investigator doesn't know.Pre-ingestion check: detect encryption before upload. Return clear error: "File is encrypted — please provide decrypted version."
Non-standard CSV encodingspandas throws UnicodeDecodeError on GBK, Windows-1252, or ISO-8859-1 encoded bank statements.Detect encoding with chardet before parsing any CSV. Normalise to UTF-8 at ingestion.
Inconsistent date formatsDD/MM/YYYY from one bank, MM/DD/YYYY from another, epoch timestamps from transaction logs. Chronology breaks.Normalise all dates to ISO 8601 at ingestion. Log original format in metadata. Never assume format.
Garbled entity names from OCR"Rajesh Kumar" becomes "R@jesh Kum4r" in a scanned KYC. Entity resolution fails. Graph node phantom.OCR quality threshold: flag chunks where character confidence drops below 80%. Route to human review before entity extraction.
Inconsistent column headers across bank statement providers"Amount" vs "Debit" vs "CR/DR" vs "Transaction Value" — same data, different labels across providers.Build a column normalisation map per provider. Maintain it as a config file, not hardcoded logic.
Handwritten annotations on documentsOCR accuracy drops to 40–60% on handwriting. Worse on Indian scripts.Flag handwritten content separately. Route to human transcription. Never auto-ingest without QC.
Build a Rejected Documents Log

Every case should have a rejected_documents log — every file that failed ingestion and exactly why. This serves two purposes: (1) the investigator knows their evidence is incomplete; (2) you have a systematic record of preprocessing failures to fix in the next sprint.

Rubber-Stamp Detection

An investigator approving 95%+ of findings without edits and with review times under 10 seconds per section is rubber-stamping — not reviewing. elapsed_review_time below a minimum threshold (suggest 30 seconds per section) triggers a quality review flag. This makes the human gate architecturally meaningful rather than ceremonial.

Cluster D — Advanced Reasoning

In Plain English

The reasoning engine. Forms a hypothesis, searches for evidence that tests it, challenges its own reasoning, revises, repeats. A 340-page document set that takes 3 days manually produces a draft finding in 90 seconds.

The Adversarial Critic — Active Hypothesis Invalidation

The Problem This Solves

A reasoning loop that only refines toward a conclusion — without genuinely testing alternative explanations — is not an investigation tool. It is a confirmation bias engine. The Adversarial Critic is the architectural control that prevents this.

For every iteration where the evidence score exceeds 0.70, the Adversarial Critic fires an explicit counter-query: what evidence, if present in the corpus, would contradict or explain away the current hypothesis? It retrieves that counter-evidence and forces the reasoning loop to account for it before proceeding.

Query TypeWhat It AsksWhat It Surfaces
Alternative explanationWhat legitimate business reason exists for the observed pattern?Legitimate invoices, documented approval chains, regulatory filings
Counter-entityDoes any evidence suggest a different entity — not the subject — is responsible?Third-party records, alternate beneficiary documents, entity resolution edge cases
Timeline invalidationIs there evidence contradicting the proposed chronology?Timestamp mismatches, document dating anomalies, contradicting records
Typology mismatchDoes the pattern match any known non-suspicious pattern from your institutional corpus?Cluster F holdout patterns, industry-standard profiles, regulatory safe harbour descriptions

A hypothesis that cannot withstand the Adversarial Critic is flagged as "contested hypothesis — alternative explanation not ruled out" and escalated with both supporting and counter-evidence presented side by side for the investigator to adjudicate.

How to Build It

  • Fire after every reasoning iteration where evidence score exceeds 0.70
  • Use same retrieval mechanism as main loop — RAG query against case namespace
  • Pass adversarial query results to Critic alongside supporting evidence
  • Block evidence score from exceeding 0.85 until adversarial check passes or investigator explicitly overrides
  • Prompt approach: "Given the current hypothesis [X], retrieve all evidence in the case corpus that could support an alternative explanation. Focus on: legitimate business context, third-party responsibility, timeline contradictions."

Prompting Principles

1

Specify Inputs and Outputs Exactly

Don't say "build a chunker." Say "Build a chunker that takes a string and returns a list of dicts with keys: chunk_id (UUID), text (string), page_ref (int), source_doc (string), char_count (int). Use 500-char chunks with 100-char overlap."

2

Include the Non-Negotiable Constraint

Always add the architectural constraint: "The namespace_id must match the case_id — never query across namespaces" or "Temperature must be 0 — no randomness in extraction tasks."

3

Ask for Tests Alongside Code

Add "also write 3 pytest test cases" to every code request. Test cases clarify what the code should do and catch errors before production.

4

Review Before Running

Read the generated code before running in production. You don't need to understand every line — but you should understand what it does. Ask Claude to explain anything unclear.

Frequently Asked Questions

Do I need to know how to code to build this?▶

You need at least one person who can write Python and work with APIs. With Claude Code or Cursor, a single developer can build Profile 3 in 8–12 weeks. You as the founder provide the domain knowledge — what the system needs to do, what constitutes a finding. AI tools handle most of the actual code writing.

Can I use cloud services instead of self-hosting everything?▶

Yes, with a critical caveat: investigation data is sensitive. The vector database, graph database, audit log, and PII token map must stay on-premise or in a jurisdiction-compliant private cloud. Only pseudonymised data may reach external AI APIs (Cluster E). For Profiles 1–4, everything can run on a private cloud VM without any external AI.

How long before the system delivers value?▶

Phase 1 (Mandatory Base) delivers defensible audit-logged case management in 4–6 weeks — before any AI reasoning is active. Phase 2 delivers evidence search in under 1 second by week 14. Phase 4 delivers draft findings in 60–90 seconds. You do not need the full platform to start getting value.

What is the minimum hardware I need?▶

Profiles 1–3 (no 70B model): cloud VM with 8GB RAM and 4 vCPUs (~$30–50/month on AWS/GCP). Profile 4+ (70B model): GPU instance with at least 40GB GPU RAM. AWS g5.12xlarge or GCP a2-highgpu-1g. Budget $150–400/month for compute. Alternatively, an RTX 4090 workstation (24GB VRAM) can run 70B in two instances.

Is the system admissible in court or regulatory proceedings?▶

The architecture is designed for maximum defensibility — tamper-evident audit chain, human approval at every finding, evidence citations on every claim. With Cluster G active, integrity is independently verifiable. However, admissibility is a legal question that depends on jurisdiction and proceeding type. Engage qualified legal counsel before relying on IRE outputs in any formal proceeding.

How do I handle Indian regulatory requirements (RBI, PMLA, SEBI)?▶

The architecture is jurisdiction-agnostic — map your specific regulatory framework into the investigation methodology document. For RBI/PMLA: your STR templates, threshold amounts, and filing windows become the query template library (A5) and framework coverage checklist (D4). For data residency: all data stores must be on Indian infrastructure — use AWS Mumbai, Azure India, or domestic providers (Jio Cloud, Tata TCS). Engage a PMLA-specialist counsel to sign off before production.

What is the difference between institutional-grade and independently verifiable audit integrity?▶

Institutional-grade (Base B5+B6): the audit chain is tamper-evident — any modification breaks the cryptographic links, detectable by anyone with access to your systems. Court-grade (+ Cluster G): the daily chain hash is anchored to an independent external ledger. A third party can verify integrity without accessing your systems. "We say it wasn't altered" vs "a third party can prove it wasn't altered."

When should I NOT build this?▶

Don't build this if: fewer than 5 investigations per month (overhead exceeds value at that scale); investigation methodology is not documented (nothing to encode); you cannot allocate one developer for 3+ months. Start with a simpler case management system and return to IRE when volume and complexity justify it.

Investigator UX — Decision Screens

The backend is only half the system. This is where product success actually happens — the investigator-facing interfaces that make the architecture usable in practice.

The Missing Piece

Both the whitepaper and most technical architectures are backend-heavy. But investigators interact with screens, not APIs. If the decision screens are confusing, slow, or disconnected from the evidence, investigators will bypass them — defeating the human gate entirely.

The Five Critical Screens

Screen 1 — Entity Resolution Decision Screen

Appears when Tier 2 or Tier 3 entity match is detected. The most important HITL screen in the system — graph quality lives or dies here.

Left PanelRight PanelBottom Bar
Entity A — all known attributes, source documents, frequency of appearanceEntity B — all known attributes, source documents, frequency of appearanceThree buttons: MERGE (green) / KEEP SEPARATE (gray) / LINK AS RELATED PARTY (yellow). Confidence score shown prominently. Timer visible.

Design principle: Never show just names. Show the evidence that led to the match proposal — field comparison table, similarity scores, co-occurrence in documents. The investigator is making an evidentiary decision, not a clerical one.

Screen 2 — Finding Review Screen (Section-by-Section)

The primary review interface. Appears after report generation for each section.

Left Panel — FindingRight Panel — Citation Map
Full finding text with inline citation tags highlighted. Edit button. Word count. Iteration count (how many reasoning cycles produced this).Every citation expanded: document name, page number, chunk text preview. Click to open source document at exact location.

Design principle: The investigator should never have to leave this screen to verify a citation. Everything needed to approve or reject is visible in one view.

Screen 3 — Contradiction Alert Screen

Appears when Cluster B detects a statement-evidence contradiction. Requires separate approval before entering the record.

Top Half — StatementBottom Half — Contradicting Evidence
Exact transcript quote with speaker ID, timestamp, session number, and interview context.The contradicting document chunk, highlighted to the exact contradicting sentence, with source and page.

Design principle: Both sides presented simultaneously. Investigator must explicitly assess the contradiction — not just note that it exists. Approval options: MATERIAL CONTRADICTION / MINOR DISCREPANCY / NOT A CONTRADICTION (with rationale required for the last option).

Screen 4 — Graph + Evidence Unified View

The most complex screen. Shows entity network alongside the documents that support it.

  • Graph panel (left 60%): Interactive entity network. Click any node → documents referencing that entity appear in right panel. Click any edge → transactions supporting that relationship appear.
  • Document panel (right 40%): Scrollable evidence feed filtered to selected node/edge. Each chunk shows source, page, date, and citation count.
  • Timeline bar (bottom): Drag to filter both graph and documents to a date range.

Design principle: Graph and documents are not separate tabs — they are linked views. Selecting a graph element immediately updates the document panel. This is the screen that makes network intelligence accessible to non-technical investigators.

Screen 5 — Adversarial Review Screen

Appears when the Adversarial Critic flags a contested hypothesis.

Left Column — Supporting EvidenceRight Column — Counter-Evidence
Evidence supporting current hypothesis with citations and evidence score.Evidence supporting alternative explanation with citations and source.

Investigator must explicitly choose: HYPOTHESIS STANDS / REVISE HYPOTHESIS / ESCALATE FOR SENIOR REVIEW. Cannot skip this screen when adversarial flag is raised.

Build Approach for Non-Technical Founders

1

Wireframe First with Claude

Describe each screen to Claude: "Build an HTML mockup of a finding review screen with: left panel showing finding text with highlighted citations, right panel showing citation map with source preview, approve/edit/reject buttons at the bottom." Claude will generate a working HTML prototype in one pass.

2

Test With a Real Investigator Before Coding the Backend

Get an HTML mockup in front of an actual investigator (even yourself) before connecting it to the backend. The most expensive UI bugs are architectural — discovered after the backend is built.

3

Graph Visualisation: Use an Existing Library

Do not build a graph renderer from scratch. Use: vis.js Network (free, straightforward) or Cytoscape.js (more powerful, still free). Connect to your Neo4j query layer via a simple REST API. The library handles the rendering.

4

Every Approval Writes to Audit Chain

The UI is the interface to the audit chain. Every button click that constitutes an investigator decision must write an immutable log entry. This is not optional — build it into the frontend as a requirement, not an afterthought.
